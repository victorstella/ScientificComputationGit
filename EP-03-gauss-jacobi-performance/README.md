# Performance Analysis — Gauss-Jacobi, Naive vs. Optimized

Solves `Ax = b` with the Gauss-Jacobi method, implemented twice — a
straightforward version and a cache/SIMD-aware version — and benchmarks
both with [LIKWID](https://github.com/RRZE-HPC/likwid) hardware performance
counters (runtime, L2/L3 cache traffic, FLOPS) across a range of system
sizes.

## Theory

**Gauss-Jacobi** is an iterative method for `Ax = b`. Like Gauss-Seidel
(see [EP-02](../EP-02-gauss-seidel-linear-systems/)), it isolates each
`x_i` from its own equation:

```
x_i^(k+1) = (b_i - sum_{j != i} A[i][j] * x_j^(k)) / A[i][i]
```

The key difference from Gauss-Seidel: every `x_i^(k+1)` is computed purely
from the **previous** full iteration `x^(k)` — never from values already
updated in the same sweep. That makes convergence somewhat slower, but
means all `n` updates in a sweep are independent of each other: each `x_i`
can be computed in any order, or in parallel, without changing the result.
That independence is exactly what the optimized implementation here
exploits.

This assignment is not about the method itself but about how much
low-level implementation choices affect performance for a fixed
algorithm and fixed number of iterations (`MAXIT`, hardcoded to 50 in
`libGaussJacobi.h`).

## Optimizations implemented

- **Linear system allocation.**
  _Naive_ (`gaussJacobi.c`): the classic array-of-pointers matrix — `n`
  separate `malloc`s, one per row, each row a `malloc`'d vector of length
  `n`. Rows can end up scattered across memory.
  _Optimized_ (`gaussJacobiOpt.c`): a single contiguous allocation for the
  whole matrix, with each row pointer computed as an offset into it. Row
  length is padded to `n + 1` whenever `n` is a power of two
  (`PAD(n)` in `gaussJacobiOpt.c`, `isPot2()` in `utils.c`) specifically to
  avoid cache-thrashing: with a power-of-two row stride, entries that are
  `n` elements apart (e.g. the same column across consecutive rows) tend
  to map to the _same_ cache set, causing excessive evictions; padding by
  one element breaks that alignment.

- **The Gauss-Jacobi sweep itself.**
  _Naive_: walks row by row, accumulating the off-diagonal sum for one
  `x_i` at a time, matching the textbook formula directly.
  _Optimized_: processes 4 rows at a time. Because `x_i^(k+1)` must not use
  any `x_j^(k+1)` computed earlier in the _same_ sweep (true even in the
  naive version, since Gauss-Jacobi always reads from the previous
  iteration), the loop accumulates the diagonal term separately and adds
  it back in (`+ A[i][i]*x_atual[i]`) instead of skipping it with an `if`
  branch — trading a branch per inner-loop iteration for one extra
  multiply-add per row, which vectorizes and pipelines better. The four
  parallel accumulators (`soma[0..3]`) let the compiler interleave
  independent floating-point operations instead of stalling on a single
  dependency chain. `x_atual`/`x_prox` are swapped by pointer at the end
  of each sweep instead of copying the array.

## How it works

`gaussJacobi-likwid.c` is the entry point:

1. Reads a linear system (`n`, matrix `A`, vector `b`, initial guess `x`)
   from a file generated by `geraSL.c`.
2. Runs `gaussJacobi()` (naive), then rewinds the input and runs
   `gaussJacobiOpt()` (optimized), each wrapped in `LIKWID_MARKER_START` /
   `LIKWID_MARKER_STOP` regions (`"GaussJacobiRegular"` /
   `"GaussJacobiOpt"`) so LIKWID can attribute hardware counters
   specifically to the solver loop — allocation, I/O, and teardown are
   excluded from the measurements.
3. Each variant writes its solution vector to its own output file.

`geraSL.c` generates a random, diagonally-dominant `n x n` system (a sign
is chosen per row so the diagonal dominates the sum of the row's absolute
values, guaranteeing Gauss-Jacobi convergence) together with a known
solution vector, so results can be checked against ground truth.

`confere_opt.sh` automates the full benchmark: for a list of system sizes,
it generates a system, runs the solver under `likwid-perfctr` with the
`L3`, `L2CACHE`, and `FLOPS_DP` counter groups, and appends the parsed
metrics to `dados.csv`, ultimately calling `plot.gp` (gnuplot) to chart the
results.

## Building and running

```sh
make avx        # builds gaussJacobi-likwid with AVX2 + native-arch flags
./gaussJacobi-likwid <system_file>
```

or run the full benchmark sweep directly:

```sh
./confere_opt.sh
```

### Requirements

- [LIKWID](https://github.com/RRZE-HPC/likwid), for the `likwid.h` header,
  `liblikwid`, and the `likwid-perfctr` CLI used by `confere_opt.sh`. The
  `Makefile` hardcodes the install path `/home/soft/likwid` — update
  `LIKWID` in `Makefile` to match your system.
- `gnuplot`, if you want `confere_opt.sh` to render `plot.gp` at the end.
- A CPU supporting AVX2 (`make avx` passes `-mavx2 -march=native`); use
  plain `make` (without the `avx` target) to build without those flags.

### Notes on the benchmark data

The numbers in `dados.csv` / the resulting plots cover only the solver
loop itself — I/O, allocation, system generation, and memory teardown are
excluded, as noted above. They were collected on:

- CPU: Intel(R) Core(TM) i5-6200U @ 2.30GHz (Skylake), 1 socket, 2 cores,
  2 threads/core.
